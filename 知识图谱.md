- 模块仍以原厂 `fan_mode`、PowerKeeper 的许可逻辑为基础。

## 3. 模块功能知识图谱

### 3.1 Hook 目标

| 目标包 | 主要职责 | 已知 Hook 方向 |
|---|---|---|
| `com.android.systemui` | 控制中心风扇模式二级菜单 | `CoolingFanController`、`CoolingFanDetailAdapter`、模式条目、点击处理 |
| `com.miui.powerkeeper` | 官方风扇策略与最终许可 | `FanStateHandler`、状态读取与模式策略衔接 |
| `com.android.settings` | 常规设置页面 | 风扇模式项和偏好同步 |
| `com.xiaomi.misettings` | 小米设置页面 | 自定义模式、按应用配置、搜索与偏好页兼容 |

模块 Xposed 元数据中曾确认的 scope：

```text
com.android.settings
com.miui.powerkeeper
com.android.systemui
com.xiaomi.misettings
```

### 3.2 SystemUI 模式条目

SystemUI 内部新增条目使用稳定 identity：

| identity | 模式 | 模块值 | 说明 |
|---:|---|---:|---|
| `4` | 狂暴模式 | `4` | 复用官方可识别模式值 |
| `5` | 自定义模式 | `5`（UI identity） | 底层以模块自定义状态维持，常见官方 `fan_mode=-1` |

关键实现流程：

```text
CoolingFanController 构造
  → ensureSystemUiModeItems()
  → 向 _secondaryItems 追加扩展条目

CoolingFanDetailAdapter.updateItems()
  → 确保条目存在
  → 屏蔽 PowerKeeper 短暂的桥接 fan_mode=2 闪烁
  → 由原始模型重新生成二级菜单内容

onDetailItemClick()
  → 识别 identity
  → 写入对应模式/偏好状态
  → 保留原生点击流程或执行模块分支
```

### 3.3 PowerKeeper 与最终许可

观察结论：

```text
target_level > 0
≈ 能观察到的官方最终风扇许可条件
```

已在只读状态观察中确认：

- `FanStateHandler.handleMessage` 与 `FanStateHandler.J("target_level", value)` 是理解原厂状态变化的重要观察点；
- 在当时的 PWM 模式试验状态下，曾捕获 `target_level=1` 与 `officialAllowed=true`；
- 原厂 PowerKeeper/HAL 能在外部 close-only 请求后再次按自身策略写入风扇相关节点。

因此必须区分：

```text
模块请求归零 / 退出模式
≠ 永久禁止原厂风扇策略
```

## 4. 反编译与逆向经验

> `reverse/` 是后续维护的核心资料，清理时明确保留。

### 4.1 逆向资料的用途

反编译材料用于回答以下问题：

```text
类名是否存在？
方法和字段在目标 HyperOS 版本中的真实名称是什么？
SystemUI 的详情页由谁持有、谁绘制？
资源名、string、drawable 归属哪个 APK？
PowerKeeper 最终通过哪个状态字段控制风扇？
```

推荐顺序：

```text
1. 从 APK / 反编译目录找类、资源和布局；
2. 在真机通过 LSPosed probe 日志确认类/方法/字段可用；
3. 只 Hook 已验证的目标；
4. 修改后构建、安装、重启对应进程；
5. 从 logcat 验证实际运行路径。
```

### 4.2 SystemUI 资源表与跨 APK 资源限制

最重要的经验：

```text
QSDetailContent.Item.iconRes 是 SystemUI 自身资源表中的整数 ID。
模块 APK 的 R.drawable ID 不能直接填给 SystemUI 的 iconRes。
```

原因：

```text
相同的整数资源 ID 在不同 APK 的 Resources 表中含义不同。
SystemUI 若用自身 Resources 解析模块的 ID，可能得到错误资源、空资源或异常。
```

已确认的原厂相关资源名包括：

```text
ic_qs_coolingfan_mode_performance
ic_qs_coolingfan_mode_silent
ic_qs_coolingfan_mode_selected
ic_qs_coolingfan_enable
quick_settings_coolingfan_mode_high
```

### 4.3 控制中心图标改造尝试与失败结论

#### 原因定位

狂暴模式、自定义模式与“高速强冷”图标重复的直接原因：

```kotlin
context.resourceId("ic_qs_coolingfan_mode_performance", "drawable")
```

扩展模式建模和 fallback 都复用了这个 SystemUI drawable。

#### 已尝试方案

1. 在模块中创建两张 24dp 单色 VectorDrawable：
   - 狂暴：风扇轮廓 + 闪电；
   - 自定义：三条滑杆 + 调节圆点。
2. 在 `createDetailView()` 返回后：
   - 用 `createPackageContext(TARGET_PACKAGE, CONTEXT_IGNORE_SECURITY)` 加载模块 drawable；
   - 遍历 `TextView`，按“狂暴模式”“自定义”定位行；
   - 找到行内第一个 `ImageView` 后调用 `setImageDrawable()`。
3. 加入日志并在真机验证。

#### 真机日志结论

```text
SystemUI fan-mode icon decoration after bind:
extreme=false
custom=false
root=com.android.systemui.qs.QSDetailContent
titles=[]
```

结论：该 HyperOS 版本中，`QSDetailContent` 在 Hook 时是**模型/容器对象**，不是实际承载可见行文本和图标的视图树。即使 `post {}` 延后一次，容器中仍没有标题。

因此下列推断不可靠：

```text
createDetailView() 返回的 QSDetailContent
= 当前实际显示的所有模式行容器
```

后续若重做图标，应优先：

```text
- 反编译追踪 QSDetailContent.Item 的最终 adapter / RecyclerView / 真实 row inflater；
- 或在实际 row bind / setItem / onBindViewHolder 类似位置 Hook；
- 先打印真实绘制容器的 class、id、child tree，再修改；
- 不要再只从空 QSDetailContent 向上猜父层级。
```

该图标尝试的源码已回滚，不在当前基线内。

### 4.4 反射与 Hook 稳定性经验

- 不同 HyperOS 版本的字段、方法可能混淆或变动；所有反射目标都应在安装 Hook 前 probe。
- 使用 `runCatching` / `onFailure` 记录失败，避免一个非关键 UI Hook 使 SystemUI 崩溃。
- 针对集合字段（例如 `_secondaryItems`），只追加自己的 identity，不改写官方 identity。
- 对模式中转值需谨慎：PowerKeeper 短时 `fan_mode=2` 可能只是退出路径桥接状态，直接呈现在 UI 会导致模式闪跳。
- 每次重装模块后通常需重启对应进程；SystemUI 验证可杀掉 `com.android.systemui`，由系统拉起新进程。

## 5. MiCharge HAL 与 PWM 研究历史（已冻结/不继续）

> 本节记录历史结论，防止未来重复踩坑。用户当前不要求无级 PWM，禁止自动恢复这些功能。

### 5.1 研究目标与结论

曾尝试通过 Xiaomi MiCharge HAL 查找 `pwm_duty` 类路径或官方控制接口。

结论：原厂 HAL node map 不支持该路径。以下操作有明显风险：

```text
字符串替换
node map 注入
code cave
vtable patch
overlay / 修改 vendor HAL
```

风险包括：

```text
/vendor/bin/hw/vendor.xiaomi.hardware.micharge-service 崩溃
AIDL 服务异常
HAL 健康损失
```

**长期约束：不得 patch、overlay 或改造 MiCharge HAL。**

已知正常 HAL SHA-256：

```text
b9a9e06c4dffd74e6d27b5062ce22fae4ee950175351043d184ed885d9c182a3
```

### 5.2 曾部署的 close-only helper（历史）

历史上曾存在 root helper 和 close-only 退出桥：

```text
SystemUI
  → ContentResolver.call()
  → PwmCloseProvider
  → root helper FIFO
  → SET <generation> 0 0
  → pwm_duty=0
```

之所以从 receiver 改为 Provider，是因为 HyperOS Greezer 拒绝了 SystemUI 指向模块 receiver 的显式广播：

```text
BroadcastQueue: Greezer Denial
```

加固后 helper 只接受：

```text
SET n 0 0
STATUS
```

历史 hardened helper SHA-256：

```text
4afb17422d448d719ada04e737bd3e972ef2d4588bc032c9cd20e54ee8f49f7e
```

该闭环曾回归成功，但它不意味着永久禁止原厂风扇策略。相关工作区文档和实验目录已经清理；真机上是否仍残留 helper 应仅在用户明确要求清理时单独处理。

### 5.3 为什么没有继续正 PWM

主要阻断因素：

1. 无法证明调用方具备独占、可信的正 PWM 写入资格；
2. 原厂 PowerKeeper/HAL 可能同时写同一节点；
3. 模块、Settings、SystemUI、root helper 之间缺少可证明的安全授权根；
4. 修改原厂 HAL 风险不可接受；
5. 用户最终明确要求停止无级 PWM 并恢复此前源码。

因此当前原则为：

```text
不实现、不测试、不部署正 duty。
不恢复 PwmCloseProvider / PwmCloseReceiver。
不恢复 GRANT / RENEW / lease / capability 协议。
不读取或写入 PWM sysfs 节点，除非用户重新明确授权并单独立项。
```
